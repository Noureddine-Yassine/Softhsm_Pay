Initialisation MK — Choisir la méthode

T0 — Générer la MK (TRNG)

Génère une MK 256 bits via RAND_bytes (TRNG noyau + hardware entropy) et la stocke chiffrée (AES-256-GCM) dans mk.bin. La MK n’est jamais affichée. Requis avant T1.

● Statut mk.bin : inconnu — cliquez Vérifier

T1 — Générer les parts SSS (optionnel)

Divise la LMK en 3 parts Shamir (GF(28), seuil 3/3). Génère lmk_share_1.sss, lmk_share_2.sss, lmk_share_3.sss — un fichier par administrateur. Bloqué si mk.bin absent.

⚠ Distribuer une seule part à chaque administrateur. Ne jamais stocker les 3 parts ensemble.

📄 lmk_share_1.sss — Admin 1
📄 lmk_share_2.sss — Admin 2
📄 lmk_share_3.sss — Admin 3

Reconstruire la LMK depuis les 3 fichiers SSS

Sélectionnez les 3 fichiers .sss distribués à chaque administrateur. La LMK est reconstituée temporairement, fragmentée (P1⊕P2⊕P3) en mémoire, puis immédiatement effacée. Aucun fichier lmk.bin n’est créé.

📄 lmk_share_1.sss — Admin 1 —
📄 lmk_share_2.sss — Admin 2 —
📄 lmk_share_3.sss — Admin 3 —
● Sélectionnez les 3 fichiers SSS pour activer la reconstruction

Fragments P1 / P2 / P3

À chaque opération crypto (EMV, PIN, MAC…), le HSM recompose brièvement la LMK (P1⊕P2⊕P3) puis mute les fragments. La référence LMK reste stable ; les empreintes P1/P2/P3 changent — cliquez Actualiser.

Coffre clés (aperçu KCV)

Clés chiffrées sous LMK dans le HSM. Le Switch stocke les cryptogrammes GCM via synchronisation.
